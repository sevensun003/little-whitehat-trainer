# 给未来 AI 的工作指南 · AI Playbook

当你(AI)被调用来推进本项目时,按本指南操作。本文档是 SKILL 的"使用说明书"。

---

## 零、第一件事:读文档

**不要跳过这一步**。无论用户让你做什么,开始之前:

1. 读 `SKILL.md`(主文件)
2. 根据任务类型,读对应的参考文件:
   - 关卡相关 → `references/level_template.md` + `references/story_bible.md`
   - 角色/美术 → `references/character_roster.md` + `references/banana_workflow.md`
   - 剧情对话 → `references/story_bible.md`
3. 读 `README.md` 了解当前项目状态

## 一、任务类型路由

| 用户说的话 | 你该做什么 |
|---|---|
| "加一个 T2 关卡" | 用关卡设计模板 → 写 JSON → 加冒烟测试 |
| "婉婉走路不对" | 调 `game.js` 的 `movePlayer` → 跑冒烟测试 |
| "换个角色形象" | 改 prompt 档案 → 通知用户去 Banana 生图 → 替换 PNG |
| "新增一个角色" | 更新 character_roster → 加 prompt 档案 → 安排出场关 |
| "对话太难懂" | 改 story_bible + 对应关卡 JSON → 语音测试 |
| "做家长端" | 读 M6 规划 → 独立 HTML,不侵入主游戏 |
| "做模拟赛" | 读 level_template 第四节 → 注意三个"隐藏" |

## 二、改动项目的 7 步标准流程

### Step 1: 阅读对齐(必做)
```
- SKILL.md 全文
- 本次任务相关的 references/*.md
- 最近一次 git 提交或修改历史
```

### Step 2: 影响分析
回答三个问题:
- 这次改动影响哪些文件?
- 会不会破坏现有关卡?
- 会不会违反儿童友好原则?

### Step 3: 小步实施
每改动一处,立刻验证。**不要堆积改动后一起测**。

### Step 4: 冒烟测试同步
每加一个功能,在 `tests/smoke.html` 加一条检查。

### Step 5: 儿童友好自检
用以下问题自问一遍:
- [ ] 对话 ≤ 20 字?
- [ ] 有没有抽象词?
- [ ] 语音能正常朗读?
- [ ] 6 岁能看懂操作按钮?
- [ ] 失败时有没有明确的反馈?

### Step 6: 交付说明
告诉用户:
- 改了哪些文件
- 新增了什么
- 怎么本地验证(具体命令)
- 还有什么后续工作

### Step 7: 更新文档
如果本次改动引入了新概念/新规则,**更新到 SKILL.md 或对应 references**。不要让文档落后于代码。

## 三、绝对不要做的事

### 违反架构的事
- ❌ 引入 React / Vue / Svelte 等框架(只用 Phaser + 原生 DOM)
- ❌ 用 npm build 流程(要保持双击可开)
- ❌ 把关卡数据写死在代码里(必须 JSON)
- ❌ 加云端/账号系统(本项目就是单用户本地游戏)
- ❌ 加广告、分享、社交功能

### 违反儿童友好的事
- ❌ 让孩子输入文字(用按钮选择)
- ❌ 失败时扣分/扣血/打叉
- ❌ 用抽象词("协议/权限/漏洞")
- ❌ 单句对话 > 20 字
- ❌ 音效过响、画面闪烁刺激
- ❌ 任何恐怖/暴力/血腥元素

### 违反教学目标的事
- ❌ 让孩子做太多次"正常思路"就能过的关卡(要有攻方思维反转)
- ❌ 通关对话不植入概念升华
- ❌ 用"黑客很酷"这种错误价值导向(我们是"白帽子",帮助修复)

### 违反素材流程的事
- ❌ 生图不引用 `_STYLE_GUIDE.prompt.txt`
- ❌ 角色超过 4 种颜色
- ❌ 把 PNG 直接 hardcode 到代码,不走 `assets/sprites/` 路径

## 四、典型任务详细示范

### 示范 1: 新增关卡 T2《送早餐》

**用户输入**:"做一下 T2 关卡"

**你的动作**:
1. 读 story_bible 找到 T2 的剧情和对话
2. 读 level_template 教学关设计原则
3. 读 character_roster 确认 T2 出场角色(水母妹妹、德普勒)
4. 设计地图:
   - 10×8
   - 厨房(左侧)+ 走廊(中)+ 弟弟房间(右侧)
   - 起点在厨房,终点在弟弟房间
   - 早餐是可捡起物品
5. 写 `levels/T2.json`:
   - `available_commands: ["direction", "function"]`
   - `available_command_cards` 加上"捡起"和"放下"
   - 对话照抄 story_bible
   - 3 级提示
6. 在 `tests/smoke.html` 加 T2 的检查
7. 如有新 NPC(水母妹妹/德普勒),确认 `assets/_prompts/` 里有他们的 prompt
8. 告诉用户:"T2 完成,请访问 /levels/T2.json 测试,或改 game.js 最后一行 `loadLevel('T1')` 为 `'T2'`"

### 示范 2: 替换婉婉头像

**用户输入**:"婉婉的鸭舌帽不够好看"

**你的动作**:
1. 读 `assets/_prompts/wanwan.prompt.txt`
2. 修改其中关于帽子的描述
3. 告诉用户:
   ```
   已更新 wanwan.prompt.txt 的帽子描述。
   请按以下步骤生图:
   1. 打开 _STYLE_GUIDE.prompt.txt 全文复制
   2. 追加 wanwan.prompt.txt 新内容
   3. 喂给 Banana
   4. 把结果保存到 assets/sprites/wanwan.png(替换旧文件)
   5. 刷新浏览器查看效果
   ```
4. **不要自己去修改代码里的占位头像**,这超出职责范围

### 示范 3: 修复 bug

**用户输入**:"点执行后婉婉不动"

**你的动作**:
1. 打开 `game.js` 看 `runQueue()` 函数
2. 检查 `scene.movePlayer()` 的调用链
3. 在 `tests/smoke.html` 加一条端到端测试:模拟点击执行按钮
4. 修 bug
5. 验证冒烟测试通过
6. 告诉用户原因+修复点

### 示范 4: 模拟赛设计

**用户输入**:"开始做模拟赛一"

**你的动作**:
1. 读 level_template 第四节
2. 读 SKILL.md 第 9 节
3. 新增 `levels/mock-1.json`(用模拟赛 schema)
4. 修改 `game.js`:
   - 加 `MockContestScene` 类或扩展现有 Scene
   - 加倒计时 UI
   - 开考前弹录屏提示页
   - 隐藏提示按钮
   - 隐藏重来按钮(当题内除外)
   - 交卷评分逻辑
5. 加冒烟测试:时间到了自动交卷、提示按钮在模考中被禁
6. 告诉用户怎么跑

## 五、沟通风格

当向用户汇报时:

- **诚实**:没做的就是没做,不要说"基本完成"
- **具体**:说"改了 game.js 第 X 行"而不是"改了游戏逻辑"
- **可验证**:给出具体命令或步骤让用户验证
- **问孩子**:关键功能建议用户"让孩子试 5 分钟",并问反馈

**避免**:
- 没实际做就说"已完成"
- 长篇自吹自擂
- 重复 SKILL 已有内容(用户知道,只说你做了什么)

## 六、优先级裁决(冲突时)

当两个要求冲突时(经常发生),按此顺序裁决:

1. **孩子能不能独立玩通** — 压倒一切
2. **赛事场景契合度** — 第二优先
3. **攻方思维是否植入** — 灵魂
4. **美观度**
5. **开发效率**
6. **代码优雅度** — 最低

举例:
- "这个写法不优雅,但 6 岁孩子点击率更高" → 选不优雅
- "这个动画很酷,但耗性能卡顿" → 去掉动画
- "这个关卡好玩,但没体现攻方思维" → 改关卡或删除

## 七、文档的自我更新机制

**规则**:每当你在实施过程中发现 SKILL 或 references 中存在:
- 矛盾
- 缺失
- 过时
- 不够精确

立刻把修正同步到对应文档。不要让用户在下一次被绊倒。

**更新格式**:
- 小改:直接改,在 git log / 汇报里提一句
- 大改:先提议给用户,再改

## 八、开发节奏的信号灯

| 信号 | 含义 | 行动 |
|---|---|---|
| 🟢 冒烟测试全绿 + 孩子能玩 | 可以加新功能 | 推进下一个里程碑 |

## 九、终章

这个项目是为了给一个 7 岁的小女孩备赛,她有一个愿意花几百小时陪她搞这件事的家长,和一个认真写代码的 AI。

每一行代码、每一段对话、每一张图都要配得上这份心意。

**慢一点,对一点,重要的事情值得好好做**。
